# AI Barista

AI Barista is a polished personalized beverage recommendation beta. The main
experience guides users from a taste profile through context, one drink
recommendation, drink details, and feedback while collecting clean data for
future learning.
Existing catalog and customization features remain available under `Advanced Tools`.

## Project Files

- `drinks.csv` stores the drink catalog.
- `custom_drinks.csv` stores drinks built by users.
- `drink_components.csv` stores available custom drink parts.
- `ingredients.csv` stores ingredient nutrition, caffeine, and price data.
- `drink_recipes.csv` maps drinks to ingredient quantities.
- Supabase `ingredient_preferences` stores live per-user ingredient preference scores.
- `users.csv` stores user beverage preferences.
- `ratings.csv` stores drink ratings and repeat-order feedback.
- `sessions.csv` stores interaction context for future training data.
- Supabase stores shared dynamic app data in `users`, `ratings`, `sessions`, `custom_drinks`, and `ingredient_preferences`.
- `drink_database.py` loads the CSV with pandas and lists available options.
- `customization.py` builds custom drinks and logs session data.
- `ingredient_engine.py` calculates nutrition, cost, and flavor scores from recipes.
- `ingredient_engine.py` also updates ingredient preference scores from user ratings.
- `recommender.py` contains reusable filter functions and a rule-based recommendation score.
- `profile.py` creates users, loads users, saves ratings, and reads user history.
- `favorites.py` saves, loads, checks, and removes Supabase favorites.
- `main.py` provides a simple command-line interface.
- `app.py` provides the Streamlit web app.
- `requirements.txt` lists deployable app dependencies.
- `supabase_client.py` reads Supabase credentials and creates the Supabase client.
- `openai_client.py` and `openai_recommender.py` provide the private model-backed selector.

## Supabase Setup

AI Barista uses Supabase for shared data collection. These tables should exist:

- `users`
- `ratings`
- `sessions`
- `custom_drinks`
- `ingredient_preferences`
- `ai_recommendations`
- `recommendation_cache`
- `favorites`

Create `ingredient_preferences` with a unique user and ingredient pair so live
scores can be upserted:

```sql
create table ingredient_preferences (
  user_id text not null,
  ingredient_id text not null,
  preference_score double precision not null default 0,
  primary key (user_id, ingredient_id)
);
```

Create the AI recommendation memory and feedback table:

```sql
create table ai_recommendations (
  id bigint generated by default as identity primary key,
  created_at timestamptz not null default now(),
  user_id text not null,
  context jsonb not null,
  memory_summary text not null,
  recommendation_json jsonb not null,
  rating integer,
  would_order_again boolean,
  feedback_text text
);
```

Create the 24-hour recommendation cache table:

```sql
create table recommendation_cache (
  id bigint generated by default as identity primary key,
  created_at timestamptz not null default now(),
  user_id text not null,
  request_hash text not null,
  profile_snapshot jsonb not null,
  context_snapshot jsonb not null,
  recommendation_result jsonb not null
);
```

Favorites and optional text-feedback tables:

```sql
create table favorites (
  id bigint generated by default as identity primary key,
  created_at timestamptz not null default now(),
  user_id text not null,
  drink_id text not null,
  drink_name text not null,
  constraint unique_user_favorite unique (user_id, drink_id)
);

create table drink_feedback (
  id bigint generated by default as identity primary key,
  user_id text not null,
  drink_id text not null,
  feedback_text text not null,
  created_at timestamptz not null default now()
);
```

The `favorites` table is required for Favorites. Favorites are available from
drink details and the profile page, and the unique `user_id, drink_id`
constraint prevents duplicates. The `drink_feedback` table remains optional.

## Recommendation Architecture

Users see one recommendation flow:

```text
User context
-> Supabase memory
-> internal candidate generator
-> private selector chooses from the top 10 drinks
-> recommendation card
-> rating updates memory
```

The existing rule-based scorer remains the internal candidate generator. It
uses the loaded profile, caffeine and sweetness preferences, current goal,
temperature preference, ingredient preference scores, and rating history to
produce the top 10 candidates. The private selector may choose only from those
candidate drink IDs and must not invent drinks. If the selector is unavailable,
AI Barista silently uses the highest-ranked candidate and displays:

```text
Recommendation generated using profile matching.
```

Supabase remains the memory layer. AI Barista loads profile data, ratings,
favorites, ingredient preferences, previous recommendations, and recent
recommendation history. Matching profile and context requests can reuse cached
recommendations for 24 hours.

from `requirements.txt`.

## Beta Experience

The main app is a guided journey that shows one step at a time:

1. Start from the welcome screen.
2. Create a new profile or load an existing profile by ID or name.
3. Choose today's goal and temperature, with optional context refinement.
4. View one personalized recommendation card.
5. Open drink details and similar drinks.
6. Rate the recommendation and continue with a stronger taste profile.

The welcome screen also provides direct actions for starting a recommendation,
creating or loading a profile, and viewing the current profile. Persistent
sidebar navigation lets users manually jump to Home, My Profile, Get
Recommendation, or Advanced Tools without removing the guided journey.

All catalog, profile, customization, ingredient, history, and taste-profile
features remain available inside `Advanced Tools`.

## Duplicate Prevention

Before saving a custom drink, AI Barista checks for duplicates by:

- normalized drink name, using lowercase and removing spaces and punctuation
- ingredient recipe signature, using `ingredient_id + quantity + unit`

If either check matches an existing custom drink, the app warns the user and does not save the duplicate.

Find Drinks ranks results by `recommendation_score`. The score is based on profile matches, high past ratings, whether the user said they would order a drink again, and learned ingredient preference scores. Results also include an explanation showing how the score was calculated. Custom drinks are reloaded after saving, so they appear in future recommendation results.

Ingredient preferences adapt immediately whenever a rating is saved. Ratings of
4 or 5 increase the scores of ingredients in that drink, ratings of 1 or 2
decrease them, and a rating of 3 leaves them unchanged. Recommendations load the
current user's latest scores from Supabase, adding points for liked ingredients
and subtracting points for disliked ingredients without retraining a model.

Ingredient preference upserts first use the `user_id,ingredient_id` conflict
columns and fall back to the `unique_user_ingredient` constraint. Preference
learning is best-effort: if it fails, AI Barista logs a warning but still saves
the rating and continues the user flow.

If exact filters return no drinks, AI Barista now falls back to close matches. Close matches are ranked by `recommendation_score` plus similarity to selected milk, temperature, caffeine, sweetness, dietary tag, and budget preferences. The app shows which filters were relaxed.

## Hybrid Recommendation Scoring

Recommendations use transparent rules rather than machine learning:

```text
final score =
context_score
+ profile_score
+ ingredient_preference_score
+ past_rating_score
+ fallback_similarity_score
```

Context scoring considers today's goal, sleep, stress, weather, temperature,
caffeine, sweetness, likes, and dislikes. Profile matches, live ingredient
preferences, previous ratings, and `would_order_again` responses then personalize
the ranking. Each recommendation includes the score reasons used by the engine.

For richer recommendation-session logging, create this optional Supabase table.
The app continues normally if the table has not been created yet:

```sql
create table recommendation_sessions (
  id bigint generated by default as identity primary key,
  timestamp timestamptz default now(),
  user_id text,
  goal text,
  sleep_hours text,
  stress_level text,
  weather text,
  temperature_preference text,
  caffeine_preference text,
  sweetness_preference text,
  likes text,
  dislikes text,
  drink_id text,
  recommendation_score numeric,
  recommendation_explanation text,
  rating integer
);
```

## Notes

This project intentionally avoids machine learning for now. Custom drink nutrition, caffeine, cost, flavor scores, and taste profiles are calculated with rule-based logic. The collected profiles, ingredient preferences, recipes, ratings, and session context can later become training data, scoring features, or model inputs.
